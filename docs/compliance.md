# 合规设计说明

> 本项目把「合规」作为工程设计的一等约束，而非事后补丁。这直接对应岗位职责第 1 条「**合规**采集」，也是面试时体现职业素养的关键。

## 1. 合规红线（本项目坚守）

| 做法 | 本项目立场 |
|---|---|
| 遵守 `robots.txt` | ✅ 默认开启，被禁路径直接跳过 |
| 保守限速、不压垮服务器 | ✅ 令牌桶 ~3s/请求 + AutoThrottle |
| 只采公开数据 | ✅ 不采需登录才可见的私密内容 |
| 不采个人隐私数据 | ✅ 只保留公开的问答文本与公开计数 |
| 规模化多账号绕过 / 代理池绕封禁 | ❌ 不做。这类做法在国内有真实法律风险 |

## 2. 为什么不做「账号池 + 代理池 + 百万级」

国内已有爬虫工程师因「绕过技术保护措施 + 规模化获取数据」被追究 `非法获取计算机信息系统数据罪` / `破坏计算机信息系统罪` 的判例。核心定罪要件正是：

1. **绕过/破坏**目标的访问控制或反爬措施；
2. **规模化**地获取数据；
3. 造成对方**服务负担**或数据被**非授权使用**。

因此本项目：
- 签名模块定位为**机制理解与复现演示**，不用于规模化抓取；
- 默认小规模、单会话、遵守 robots 与限速；
- 分布式能力作为**架构展示**，强调「如何在合规前提下横向扩展」。

## 3. 合规实现映射

| 合规要求 | 代码位置 |
|---|---|
| robots 遵守 | `zhihu_crawler/compliance/robots.py` · `RobotsGate` |
| 限速 | `zhihu_crawler/compliance/throttle.py` · `TokenBucket` |
| 退避重试（避免雪崩重试压垮对方） | 同上 · `retry_with_backoff` |
| Scrapy 侧合规基线 | `scrapy_project/.../settings.py`（ROBOTSTXT_OBEY / AutoThrottle） |
| 数据校验（准确性/完整性） | `zhihu_crawler/models.py` · `Answer.validate` |

## 4. 面试话术

> "我把这个项目的合规性做进了架构：robots 遵守和限速是默认开启、无法关闭的一等公民；签名逆向我做的是机制理解和算法复现，用来展示能力，但项目默认小规模合规运行，不做账号池和代理绕封禁——因为那在国内有明确的法律风险，也不符合企业做数据合规的初衷。我认为一个数据采集工程师的价值，是在合规边界内把数据拿得又稳又准，而不是无视规则。"
