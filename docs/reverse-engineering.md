# 知乎签名 `x-zse-96` 逆向方法论

> 本文档记录对知乎请求签名机制的**逆向分析方法论**。重点是"如何分析"，而非提供一个恒久可用的绕过实现——因为签名的密钥表/字母表/版本前缀会随知乎前端 Webpack bundle 轮换。掌握方法，才能在算法变更时快速跟进。

---

## 0. 背景：知乎的请求签名体系

知乎的受保护 API（搜索、推荐、部分内容接口）会校验一组请求头/Cookie：

| 字段 | 位置 | 作用 |
|---|---|---|
| `d_c0` | Cookie | 设备指纹标识，首次访问下发 |
| `x-zse-93` | Header | 版本串（如 `101_3_3.0`），基本为常量 |
| `x-zse-96` | Header | **核心签名**，随请求 path/query 变化 |
| `x-zst-81` | Header | 部分接口的附加令牌 |

服务端用同样算法重算 `x-zse-96` 并比对，不匹配则返回错误。所以要么让浏览器带着环境算，要么把算法复现出来。

---

## 1. 第一步：DevTools 定位加密入口

**目标**：找到 `x-zse-96` 是被哪段 JS 代码写进请求头的。

1. 打开 Chrome DevTools → Network，触发一次搜索请求，确认请求头里有 `x-zse-96`。
2. 用 **XHR/fetch 断点** 或搜索关键字定位：
   - Sources 面板 → 全局搜索（`Ctrl+Shift+F`）`x-zse-96`。
   - 通常命中一处 `t.headers["x-zse-96"] = ...` 或 `setRequestHeader("x-zse-96", ...)`。
3. 在赋值处下断点，重新触发请求，命中后查看调用栈（Call Stack），逐帧上溯到**真正计算签名的函数**。
4. 常见特征：某个函数接收一个拼接串（`93 + url + dc0`），先 `md5`，再调用一个高度混淆的 `encrypt`。

> 💡 技巧：知乎 JS 经过 Webpack 打包 + 变量名混淆（`a`,`b`,`c`,`_0x...`）。断点法比纯读代码高效得多——让运行时告诉你数据怎么流动。

---

## 2. 第二步：还原被混淆的 encrypt 函数（AST 派上用场）

定位到 `encrypt` 后，它往往是这样的形态：

```js
function _0x3f2a(e){var t=_0x1b['default'](e);return _0x2c(t,_0x4d)}
// 里面充斥 _0x 十六进制变量名、字符串数组解密、控制流平坦化
```

肉眼读不动。这里就是我**自研 IDL Parser（AST 深度解析）经验**的用武之地：

### 2.1 用 AST 做反混淆

把混淆的 JS 交给 **Babel/AST 工具链**（`@babel/parser` → `@babel/traverse` → `@babel/generator`）做结构化还原：

| 混淆手段 | AST 还原策略 |
|---|---|
| 字符串数组 + 解密函数 | 遍历 AST，对 `_0xarr[idx]` 调用做**常量折叠**，把解密结果回填 |
| 十六进制变量名 | 遍历 `Identifier` 节点批量重命名为可读名 |
| 控制流平坦化（switch-while） | 分析 `SwitchStatement` 的 case 顺序，重建线性控制流 |
| 逗号表达式 / 序列压缩 | 拆 `SequenceExpression` 为多条语句 |

> 这套「解析 → 遍历 → 变换 → 重新生成」的流程，和我做 IDL Parser 时把接口定义源码解析成 AST、再做类型推导与代码生成，是**同一套心智模型**。

### 2.2 提取关键常量

反混淆后，`encrypt` 的逻辑通常暴露为：
- 一个**位运算变换**（`e[i] ^ key[i % n]` 之类）；
- 一张**乱序 base64 字母表**；
- 一个**版本前缀**（如 `"2.0_"`）。

把这三样从还原后的 AST 里读出来，就能在 Python 里 1:1 复现。

---

## 3. 第三步：Python 复现与验证

见 [`zhihu_crawler/signature.py`](../zhihu_crawler/signature.py)：

```python
plain  = f"{x_zse_93}+{path_with_query}+{d_c0}"   # 1. 拼原文
digest = md5(plain).hexdigest()                    # 2. MD5
enc    = zhihu_encrypt(digest)                     # 3. 复现的密码器
sig    = "2.0_" + enc                              # 4. 拼版本前缀
```

**验证闭环**：在浏览器 Console 里手动调用页面的 `encrypt(sameInput)`，把输出和 Python 版本逐字节对比。一致即复现成功。

---

## 4. 为什么这样做既专业又合规

1. **能力等价**：完整走通"定位入口 → AST 反混淆 → 复现 → 验证"，展示的是真正的逆向工程能力。
2. **合规边界清晰**：本项目把签名模块定位为**机制演示**，默认遵守 robots + 保守限速 + 小规模运行，不做账号池/规模化绕过——后者在国内有真实法律风险。
3. **可迁移**：AST 反混淆方法论不止用于知乎，对任何混淆前端的分析都通用。

## 5. 常见追问 & 参考答案

**Q: 签名变了怎么办？**
A: 密钥表/字母表/前缀会随 bundle 轮换，但入口定位和 AST 反混淆流程不变。我把常量抽成配置，算法变更时重跑反混淆脚本读出新常量即可，不用重写逻辑。

**Q: 为什么用 AST 而不是正则替换来反混淆？**
A: 正则处理不了嵌套结构和作用域。AST 能精确定位 `Identifier`/`CallExpression` 节点、做作用域感知的重命名和常量折叠，这是我做 IDL Parser 时的核心经验。

**Q: 直接用无头浏览器带环境跑不行吗？**
A: 可以，且更稳（`Playwright` 注入页面上下文直接调 `encrypt`）。纯 Python 复现的价值是**吞吐高、资源省**。工程上我会按场景权衡：小规模/易变接口用浏览器，稳定高频接口用复现算法。
