# 架构与分布式扩展设计

## 1. 分层架构

```
┌──────────────────────────────────────────────────────────────┐
│  入口层   run.py (单机)   │   scrapy crawl (分布式集群)          │
├──────────────────────────────────────────────────────────────┤
│  合规内核  RobotsGate · TokenBucket · retry_with_backoff        │  ← 一等公民
├──────────────────────────────────────────────────────────────┤
│  采集层   ZhihuClient (signature 签名头 → requests)             │
├──────────────────────────────────────────────────────────────┤
│  解析层   parser (JSON API 首选 / HTML 回退 · 多重选择器)        │
├──────────────────────────────────────────────────────────────┤
│  AI 清洗   ContentCleaner (ollama / openai / rule 三档降级)      │
├──────────────────────────────────────────────────────────────┤
│  存储层   Storage (SQLite 默认 / MySQL 可选)                    │
└──────────────────────────────────────────────────────────────┘
              分布式去重：BloomFilter / RedisBloomDedup
```

**设计原则**：单机版与分布式版**共享同一套领域逻辑**（models / ai / storage / signature）。Scrapy 管道直接 import 单机模块，避免重复实现——这是可维护性的关键。

## 2. 单机 vs 分布式

| 维度 | 单机版 (`run.py`) | 分布式版 (`scrapy_project/`) |
|---|---|---|
| 调度 | 顺序循环 | Scrapy-Redis 共享队列 |
| 去重 | 内存 `BloomFilter` | Redis 共享位数组 / RFPDupeFilter |
| 限速 | `TokenBucket` | AutoThrottle + DOWNLOAD_DELAY |
| 断点续爬 | 无 | `SCHEDULER_PERSIST=True` |
| 适用 | 小规模、演示、调试 | 多节点横向扩展 |

## 4. 布隆过滤器容量估算

由 `_optimal_params(n, p)`：

| URL 规模 n | 误判率 p | 位数 m | 内存 | 对比 set 存 URL(~80B/条) |
|---|---|---|---|---|
| 100 万 | 1% | ~9.6 Mbit | ~1.2 MB | ~80 MB（省 66x） |
| 1 亿 | 1% | ~958 Mbit | ~114 MB | ~8 GB（省 70x） |

> 用双哈希法（Kirsch-Mitzenmacher）只算 2 个基础哈希衍生出 k 个，省 CPU。

## 5. 可扩展方向

- **代理与会话**：`ZhihuClient` 预留了 UA 轮换，可扩展代理中间件（稳定性工程，非绕封禁）。
- **增量采集**：用 `crawled_at` + 布隆去重支持增量更新。
- **清洗异步化**：AI 清洗是 IO 密集，可拆成独立消费者进程从 Redis 取原始 item 清洗，采集与清洗解耦、各自伸缩。
- **监控**：接 Scrapy 的 stats + Prometheus 导出请求成功率/限速命中率。
